# Forge Bug: testenv-lcr does not configure containerd TLS trust in Kind

## Summary

When using `testenv-lcr` with `testenv-kind`, the container registry is set up with TLS certificates (via cert-manager), but containerd inside the Kind cluster is not configured to trust these certificates. This causes image pulls to fail with timeout errors.

## Environment

- Forge version: v0.16.0
- Kind version: v0.27.0
- Kubernetes version: v1.34.0

## Steps to Reproduce

1. Configure a test stage with `testenv-kind`, `testenv-lcr`, and `testenv-helm-install`
2. Push images to the testenv-lcr registry (works via port-forward)
3. Install a Helm chart that references images from `testenv-lcr.testenv-lcr.svc.cluster.local:5000`
4. Observe that the deployment times out waiting for pods to become ready

## Expected Behavior

Pods should be able to pull images from the testenv-lcr registry inside the Kind cluster.

## Actual Behavior

Helm install times out with `Error: INSTALLATION FAILED: context deadline exceeded`.

The root cause is that containerd inside the Kind nodes does not trust the self-signed TLS certificate generated by cert-manager for the testenv-lcr registry.

## Evidence

From forge test-all output:

```
✅ Pushed image: testenv-lcr.testenv-lcr.svc.cluster.local:5000/shaper-controller-container:latest
...
2025/11/29 13:45:09 Installing chart 2/3: shaper-controller (release: shaper-controller)
...
❌ Stage 'integration' failed: ... helm install failed: exit status 1, output: Error: INSTALLATION FAILED: context deadline exceeded
```

The image push succeeds (via port-forward to localhost:5000), but the pod inside Kind cannot pull from the registry because containerd doesn't trust the TLS certificate.

## Suggested Fix

The `testenv-lcr` engine should configure containerd in Kind to trust the registry CA certificate. This typically involves:

1. Extract the CA certificate from the cert-manager issuer/certificate
2. Configure containerd on each Kind node to trust this CA for the registry hostname
3. This can be done by:
   - Adding the CA to `/etc/containerd/certs.d/<registry-hostname>/ca.crt` on each node
   - Or configuring containerd to use insecure mode for this specific registry

## Workaround

Currently there is no workaround using forge's built-in testenv engines.

## Related Configuration

```yaml
- alias: testenv-integration
  type: testenv
  testenv:
    - engine: go://testenv-kind
    - engine: go://testenv-lcr
      spec:
        enabled: true
        namespace: testenv-lcr
        imagePullSecretName: testenv-lcr-credentials
        imagePullSecretNamespaces:
          - default
          - shaper-system
        images:
          - name: local://shaper-controller-container:latest
    - engine: go://testenv-helm-install
      spec:
        charts:
          - name: shaper-controller
            sourceType: local
            path: ./charts/shaper-controller
            namespace: shaper-system
            values:
              image:
                repository: testenv-lcr.testenv-lcr.svc.cluster.local:5000/shaper-controller-container
                tag: latest
                pullPolicy: IfNotPresent
              imagePullSecrets:
                - name: testenv-lcr-credentials
```
